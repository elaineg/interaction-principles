# Interaction Principles
Purpose: for product/UX designers, front-end engineers, and design-curious PMs studying interaction-design craft, a free no-signup interactive curriculum where every load-bearing "how interfaces should feel" concept is a manipulable demo with the underlying physics parameters exposed as honest live controls.
Problem: Designers/engineers building interaction-design taste (for a portfolio, an interview, or a job-hunt sprint) need to actually FEEL principles like 1:1 tracking, springs, interruptibility, and velocity handoff — not just read about them. They hit this in bursts (portfolio/interview prep, revisited as reference), not weekly; recurrence is honestly LOW/bursty for a learning tool. The core audience is ~2 "extremely useful" (Aisha, product designer; Marcus, front-end eng) + 2-3 "somewhat" (Rob, Sam, Elena) of a 10-persona roster — this underperforms the daily-utility recurrence gate BY DESIGN and is a forced priority-lane build, not a queue admit.
Beats alternative: The closest siblings are PAID and signup-gated — animations.dev (Emil Kowalski, ~$199, signup, video + sandbox), Devouring Details (Rauno Freiberg, paid, registration-gated), Josh Comeau courses ($150+, signup). The free options (MDN, framer-motion docs, Disney 12-principles PDFs, scattered essays/talks) are fragmented and read-only. Ours is the single hands-on synthesis that is FREE, NO-SIGNUP, and PARAMETER-EXPOSED: it gives you the spring's mass/stiffness/damping, latency, frame budget, and lift-off velocity as honest draggable instruments you feel under your finger — no incumbent does this for free.
Core flows:
1. Land + navigate — page loads already running on lesson 01 (an ink object responding to the pointer, controls visible within 5 seconds). The sticky left rail lists all 8 lessons (number + title + one-line subtitle); clicking one (or Prev/Next) switches the main column to that lesson; last-viewed lesson is restored from localStorage.
2. Read + manipulate the live demo (THE DIFFERENTIATOR) — each lesson shows a plain-English lede, a bordered demo stage you interact with via Pointer Events, and a control panel of labeled sliders/toggles/segmented controls that change the demo's parameters live, each with a tabular-nums value readout, and live instruments (gain delta, latency ms, FPS, velocity px/s, frame-budget bar) that reflect the ACTUAL rendered motion. The exposed-parameter controls ARE the reason this app exists — every lesson must have at least one control that visibly changes its demo.
3. A/B the good-vs-bad comparison — each lesson offers a side-by-side or toggle between the correct behavior and a naïve/janky one (spring vs easing, interruptible vs non-interruptible, velocity-handoff vs seam, frame-drop, continuity vs cut), with the bad/janky side and dropped-frame markers in `--red`. The demo + lede stand on their own (no "why it matters" note).

  1. Direct manipulation & 1:1 tracking — dragging the ink square tracks the pointer; the `GAIN` segmented control (0.5×/1×/2×) and `LATENCY` slider (0-200ms) change tracking (P0 gain math: object displacement = gain × pointer displacement measured from the drag anchor — 0.5× moves the object half the pointer distance, 2× twice; at GAIN=1, latency=0 the object sits 1:1 under the pointer; the px-delta + active-gain readout reflects this, ≈0 at 1× and growing at 0.5×/2×).
  2. Physics spring vs easing curve — two objects fire A→B simultaneously (left spring, right cubic-bézier); MASS/STIFFNESS/DAMPING sliders change the spring's overshoot/settle and DURATION + draggable bézier control points change the easing, and both plot position-over-time.
  3. Interruptibility & state continuity — an explicit labeled comparison of two panels animating the same open↔closed action: an `INTERRUPTIBLE` one that can be grabbed mid-flight (hands off current position/velocity, re-targets on release) and a `NON-INTERRUPTIBLE` one (in `--red`) that ignores input until the animation finishes; a shared trigger fires both so the contrast is feelable. (Supersedes the old NAÏVE RESTART ghost toggle.) 2026-06-20 CATCH ZONE: the INTERRUPTIBLE sub-stage is noticeably TALLER (longer vertical travel distance, not a slower animation) — after firing Open/Close, a stranger can press the moving INTERRUPTIBLE panel MID-FLIGHT and redirect it (catch window comfortable, not frame-perfect) on both desktop and touch, while pressing the NON-INTERRUPTIBLE panel mid-flight does NOT change its trajectory until it settles.
  4. The animation principles in UI — ONE single object performs ONE action with per-principle ON/OFF toggles (anticipation, follow-through/overlap, slow-in/slow-out, squash & stretch) that LAYER onto that same object so combined effects compose, plus an INTENSITY slider that scales all enabled effects.
  5. Velocity handoff / flick — flicking a card continues under a deceleration model seeded by lift-off velocity; the captured velocity vector + px/s readout is drawn at release (P0: the readout magnitude/direction matches the card's actual post-release motion), and the `SEAM` debug toggle produces a visible `--red` discontinuity. P0 FRICTION (2026-06-20): the `DECELERATION`/friction slider value actually feeds the per-frame decel model (read every step, dt-scaled, no stale closure) and is clearly FELT — for ONE fixed lift-off velocity, higher friction ⇒ visibly SHORTER total glide distance AND shorter time-to-rest, monotonically across the slider range (low = long far coast, high = stops quickly); the slider's label/units/direction are legible. P0 bounds (unchanged): on any flick (incl. a hard one) the card stays WITHIN the demo stage's bounding box and BOUNCES (elastic velocity reflection, restitution < 1) off the edges, decelerating to rest inside — it never escapes the stage.
  6. Latency, frame budget & jank — an FPS meter and per-frame budget bar read real `requestAnimationFrame` timestamps; the `INJECT WORK (ms/frame)` slider drops the measured FPS and stutters the motion (P0: the FPS meter shows honestly LOWER fps as injected work increases — not a faked number), and a 60HZ/120HZ switch (owned by this lesson, living inside its demo region — NOT in the global top bar) redraws the budget line at 16.6 vs 8.3ms.
  7. Feedback timing & affordances / JND — pressing the button gives an immediate touch-down inversion; the `RESPONSE DELAY` slider (0-300ms, marked JND zone) inserts confirmation latency you can feel, TOUCH-DOWN HIGHLIGHT on/off toggles the affordance, and HAPTIC on/off fires `navigator.vibrate` where supported (and says so when unavailable).
  8. Spatial / navigational continuity — tapping a grid tile expands it in place into a detail view and returns to its origin tile; the `CONTINUITY` toggle switches to a naïve cut/cross-fade, and SLOW-MO (0.25×) makes the shared-element morph legible.

Hard constraints (for builder/verifier): 100% client-side, NO backend; localStorage only for last-viewed lesson + control settings (device-local, never identity-gated). Physics is HAND-ROLLED from scratch (semi-implicit-Euler spring integrator with mass/stiffness/damping, weighted last-few-samples velocity tracker, friction/deceleration model, shared rAF loop with fixed-timestep accumulator) — the from-scratch engine IS the lesson, no animation library. All dragging via Pointer Events with `setPointerCapture`. Instrument honesty is P0: the FPS meter, velocity readout, and gain/latency deltas MUST reflect the actual rendered motion — a lying instrument is worse than none.
