# Interaction Principles
Purpose: for product/UX designers, front-end engineers, and design-curious PMs studying interaction-design craft, a free no-signup interactive curriculum where every load-bearing "how interfaces should feel" concept is a manipulable demo with the underlying physics parameters exposed as honest live controls.
Problem: Designers/engineers building interaction-design taste (for a portfolio, an interview, or a job-hunt sprint) need to actually FEEL principles like 1:1 tracking, springs, interruptibility, and velocity handoff — not just read about them. They hit this in bursts (portfolio/interview prep, revisited as reference), not weekly; recurrence is honestly LOW/bursty for a learning tool. The core audience is ~2 "extremely useful" (Aisha, product designer; Marcus, front-end eng) + 2-3 "somewhat" (Rob, Sam, Elena) of a 10-persona roster — this underperforms the daily-utility recurrence gate BY DESIGN and is a forced priority-lane build, not a queue admit.
Beats alternative: The closest siblings are PAID and signup-gated — animations.dev (Emil Kowalski, ~$199, signup, video + sandbox), Devouring Details (Rauno Freiberg, paid, registration-gated), Josh Comeau courses ($150+, signup). The free options (MDN, framer-motion docs, Disney 12-principles PDFs, scattered essays/talks) are fragmented and read-only. Ours is the single hands-on synthesis that is FREE, NO-SIGNUP, and PARAMETER-EXPOSED: it gives you the spring's mass/stiffness/damping, latency, frame budget, and lift-off velocity as honest draggable instruments you feel under your finger — no incumbent does this for free.
Core flows:
1. Land + navigate — page loads already running on lesson 01 (an ink object responding to the pointer, controls visible within 5 seconds). The sticky left rail lists all 8 lessons (number + title + one-line subtitle); clicking one (or Prev/Next) switches the main column to that lesson; last-viewed lesson is restored from localStorage.
2. Read + manipulate the live demo (THE DIFFERENTIATOR) — each lesson shows a plain-English lede, a bordered demo stage you interact with via Pointer Events, and a control panel of labeled sliders/toggles/segmented controls that change the demo's parameters live, each with a tabular-nums value readout, and live instruments (gain delta, latency ms, FPS, velocity px/s, frame-budget bar) that reflect the ACTUAL rendered motion. The exposed-parameter controls ARE the reason this app exists — every lesson must have at least one control that visibly changes its demo.
3. A/B the good-vs-bad comparison + read why-it-matters — each lesson offers a side-by-side or toggle between the correct behavior and a naïve/janky one (spring vs easing, continuity vs naïve-restart, velocity-handoff vs seam, frame-drop, continuity vs cut), with the bad/janky side and dropped-frame markers in `--red`; below it a hairline-separated 2-4 sentence "why it matters for your portfolio" note.
Success checks:
- The page loads at the prod URL and renders lesson 01 already interactive (an ink object tracks the pointer on drag) within the first view, with no signup/login wall anywhere.
- The left rail lists exactly 8 lessons; clicking a rail item or Prev/Next swaps the main column to that lesson, and reloading the page restores the last-viewed lesson (localStorage).
- All 8 lessons render and each has at least one control that VISIBLY changes its demo when adjusted:
  1. Direct manipulation & 1:1 tracking — dragging the ink square tracks the pointer; the `GAIN` segmented control (0.5×/1×/2×) and `LATENCY` slider (0-200ms) change tracking, and the px-delta + active-gain readout updates (P0: at GAIN=1, latency=0 the object sits under the pointer; the displayed delta matches the rendered offset).
  2. Physics spring vs easing curve — two objects fire A→B simultaneously (left spring, right cubic-bézier); MASS/STIFFNESS/DAMPING sliders change the spring's overshoot/settle and DURATION + draggable bézier control points change the easing, and both plot position-over-time.
  3. Interruptibility & state continuity — grabbing the animating sheet mid-flight hands off its current position/velocity and re-targets on release; the `NAÏVE RESTART` toggle ON makes it snap/restart (the bad version) so the difference is feelable.
  4. The 12 animation principles in UI — per-principle ON/OFF toggles (anticipation, follow-through, slow-in/slow-out, squash & stretch) A/B the same action and the INTENSITY slider scales the effect.
  5. Velocity handoff / flick — flicking a card continues under a deceleration model seeded by lift-off velocity; the captured velocity vector + px/s readout is drawn at release (P0: the readout magnitude/direction matches the card's actual post-release motion), the DECELERATION slider changes coast distance, and the `SEAM` debug toggle produces a visible `--red` discontinuity.
  6. Latency, frame budget & jank — an FPS meter and per-frame budget bar read real `requestAnimationFrame` timestamps; the `INJECT WORK (ms/frame)` slider drops the measured FPS and stutters the motion (P0: the FPS meter shows honestly LOWER fps as injected work increases — not a faked number), and the 60HZ/120HZ switch redraws the budget line at 16.6 vs 8.3ms.
  7. Feedback timing & affordances / JND — pressing the button gives an immediate touch-down inversion; the `RESPONSE DELAY` slider (0-300ms, marked JND zone) inserts confirmation latency you can feel, TOUCH-DOWN HIGHLIGHT on/off toggles the affordance, and HAPTIC on/off fires `navigator.vibrate` where supported (and says so when unavailable).
  8. Spatial / navigational continuity — tapping a grid tile expands it in place into a detail view and returns to its origin tile; the `CONTINUITY` toggle switches to a naïve cut/cross-fade, and SLOW-MO (0.25×) makes the shared-element morph legible.
- Chrome is strict SSENSE monochrome (1px hairlines, square corners, uppercase tracked micro-labels, no shadows/decorative color); `--red #B00020` appears ONLY on the bad/janky side of a comparison and dropped-frame markers (text/marks, never fills).
- With `prefers-reduced-motion` on, chrome transitions are dimmed and each demo canvas shows an explicit "enable motion to view this lesson" affordance rather than silently breaking.
Hard constraints (for builder/verifier): 100% client-side, NO backend; localStorage only for last-viewed lesson + control settings (device-local, never identity-gated). Physics is HAND-ROLLED from scratch (semi-implicit-Euler spring integrator with mass/stiffness/damping, weighted last-few-samples velocity tracker, friction/deceleration model, shared rAF loop with fixed-timestep accumulator) — the from-scratch engine IS the lesson, no animation library. All dragging via Pointer Events with `setPointerCapture`. Instrument honesty is P0: the FPS meter, velocity readout, and gain/latency deltas MUST reflect the actual rendered motion — a lying instrument is worse than none. Edge economy: disable Next.js Link prefetch, all heavy testing against local `next start`, touch Vercel only for the final prod deploy + tiny smoke.
Out of scope: accounts/auth/payments/Turso/any backend; sharing links; content CMS; video; mobile-only gestures beyond Pointer Events; exhaustive coverage beyond these 8 lessons (optimize for how correct and delightful each demo feels over breadth).
